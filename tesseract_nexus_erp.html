What NAM makes possible
ERP integration that learns,
because the system has memory.
Nexus ERP is a bidirectional integration platform — Sage Intacct, Dynamics 365, SAP S/4HANA, NetSuite, Oracle ERP Cloud, or any custom plugin — connected through a single AI-driven transformation layer. Every schema, every approved mapping, every drift event, every sync failure is written to NAM as an addressable record. When the agent proposes a new mapping or heals a broken one, it does so with NNLM-grounded context drawn from the platform's full operational memory. No hallucinated field names. No re-discovering what the system has already learned.
What NAM stores for Nexus ERP
ERP entity schemas schema:{connector_id}:{entity}
Every discovered schema flattened into addressable text — system, entity, field name, data type, description. Re-indexed on every discovery run. The agent retrieves these by semantic match: "what field holds the invoice total?" resolves to the right field across every ERP that has one.
Approved field mappings mapping:{pipeline_id}
When a human approves a mapping, the full source → transform → target set is written to NAM. Future proposals across any pipeline can retrieve and reuse them — every successful mapping informs the next, even between different ERP combinations.
Schema drift events drift:{diff_id}
Each detected schema change — added, removed, or type-changed fields — is indexed alongside its connector and entity context. The healing agent retrieves prior drift events when reasoning about how to recover from a similar change.
Sync failures sync_event:{pipeline_id}:{record_id}
Failed sync attempts — error message, pipeline, entity, record — are written into NAM so the healing agent has direct access to operational evidence, not just static schema definitions. Failure patterns become retrievable context.
