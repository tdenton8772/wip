NNLM — NAM-Native Language Model
Grounded Generation · Structural Citation · Enforced Rejection
The language layer
Language that is grounded by construction,
not by convention.
NNLM is not a language model with retrieval bolted on. It is a four-stage pipeline where every claim the system makes traces to a specific retrieved record, every citation is a structural output token, and rejection — saying "I don't know" — is a first-class outcome enforced by the architecture. The generation model cannot assert without evidence. It is structurally prevented from doing so.
Architectural properties
Grounding by construction
Grounding is a pipeline invariant, not a model behavior. The generation model receives only pre-filtered, pre-scored evidence. It cannot access its training knowledge freely — it operates within the boundary set by the selection output.
Citation as structural output
Citations are first-class output tokens, not post-processing. The generation model is trained to tag every factual claim with the record that supports it. Missing citations are measurable failures. False citations are identifiable.
Rejection over hallucination
When the selection stage finds insufficient evidence, the pipeline produces a structurally determined rejection — not a generated "I'm not sure." The rejection is based on concrete scores against a configurable threshold. No hallucinated response is produced.
Soft prompt injection
The selection stage passes learned embeddings — not text — to the generation model. These carry structured relevance signals: which records matter, how strongly, and the overall evidence confidence. The model is told what matters, not left to infer it.
Per-stage measurability
Every stage boundary is observable and auditable. Encoding output, retrieval candidates, selection scores, and generation citations are all inspectable. When a response is wrong, you can trace it to the exact stage that failed — not just "the model."
Independent stage improvement
Each stage has explicit input/output contracts. Better entity resolution? Improve the encoder. Missing records? Improve NAM coverage. Too much noise? Improve the selection model. No stage requires another to be retrained.

Multi-turn conversation — session memory
Context window approach — what NNLM does not do
Most conversational AI handles multi-turn interaction by concatenating previous turns into the prompt. This has hard limits: context windows fill up, earlier turns get less attention as conversation grows, the model re-processes the entire conversation at every turn, and there is no structural representation of what was discussed — just accumulated text.
Session memory — what NNLM does
Each conversation is a session with structured state — entity history, resolved references, topic continuity — backed by NAM's addressing system. The encoder resolves pronouns and coreferences against the session before generating a query plan. Downstream stages are unaware of session context — they receive a fully resolved query plan, nothing more.
Multi-turn resolution example
Turn 1
User: "What is Neural Addressable Memory?"
Encoder identifies entity: NAM. Query plan generated. Session entity history updated.
Turn 2
User: "Who created it?"
Encoder resolves "it" → NAM from session context. Query plan: "Who created NAM?" — fully resolved before retrieval.
Turn 3
User: "What are the design principles?"
Encoder carries topic continuity — NAM — forward. Query plan targets NAM design principles specifically. No prompt stuffing at any turn.
