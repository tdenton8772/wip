How addresses are built
From raw document to deterministic address
in a single streaming pipeline.
Every record ingested into NAM passes through a staged NATS pipeline where independent encoder heads extract different semantic dimensions in parallel. Each head publishes its result. The addressing worker joins all heads, bundles them per record, and writes 27–85 deterministic geometric addresses per document into the NAM address space. The encoder heads are pluggable — new dimensions can be added without modifying the pipeline.
Pluggable encoder heads
Head Workers NATS input NATS output What it extracts
NLP 1 ×1 raw parsed Rule-based parse — tokenize, POS tag, lemmatize, NER, dep parse. ~13,385 parses/sec. Zero model weights. Fully deterministic. SpaCy backend available.
Ontology 1 ×1 parsed ontology + heads.ontology Domain classification. Assigns the record's canonical ontology type — the primary coordinate axis that organises the entire address space.
Entity 6 ×6 queue group ontology heads.entity Surface form resolution — "Einstein" → entity:albert_einstein. 3-level cache hierarchy (L1 in-memory, L2 LMDB on hostPath, L3 KV store). Most KV-intensive encoder.
Attribute 1 ×1 ontology heads.attribute Adjective extraction. Encodes descriptive and qualifying properties of entities into the address coordinate space.
Affordance 1 ×1 ontology heads.affordance Verb extraction. Encodes what entities do or can do — the action and capability dimensions of the address.
Context 1 ×1 ontology heads.context Topic classification. Encodes the situational and domain context that completes the geometric address.
Addressing 1 ×1 heads.* NAM bucket Streaming join across all head results. EntityBundler.bundle() → build_from_bundles(). Writes 27–85 NDJSON addresses per record via KV APPEND. The only writer to the address space.
Entity head — 3-level cache hierarchy
L1 — in-process
Per-process dicts
Microseconds
Thread-safe dicts protected by threading.Lock(). Surface form lookups, fuzzy candidate reverse index, and _MISS sentinel for negative caching. Scope: single container process lifetime. Eliminates redundant KV calls for hot entities within a session.
L2 — node-level
LMDB on hostPath
~1–2 µs zero-copy
LMDB on RAM-backed hostPath (/dev/shm/nam-entity-lmdb). Written by a DaemonSet pod streaming the session bucket via DCP. Zero-copy mmap reads. All entity workers on the same node mount the same path readonly. Shared negative cache via __LMDB_MISS__ sentinel.
L3 — cluster-wide
Data service KV
~1–5 ms network
Persistent store in the session bucket (256 vbuckets). Type-scoped lookups with unscoped fallback. 2-phase batch fetch reduces KV gets by ~40–50%. CAS operations for concurrent entity registration — first-writer-wins. Async write thread queues background flushes.
Data service buckets
default 1024 vb
Source documents. Raw records loaded via CDC or bulk load. The DCP adapter streams mutations from this bucket into the ingestion pipeline. RocksDB-Cloud backend with optional S3 backup and sub-second restore.
nam 1024 vb
Semantic addresses. Written exclusively by the addressing worker via KV APPEND. Each address key accumulates record references as NDJSON — one line per record. The ep-engine is patched so APPEND auto-creates keys via SET, eliminating read-modify-write cycles.
graph index
Dimensional graph index. Written in parallel with nam bucket writes by the addressing worker's second async write path. Contains five key types: single-axis, dual-axis, triple-axis, entity-to-record, and record-reverse. Validated at 19.7M entries across 500K records. Enables pure KV prefix-scan traversal at query time — no floating-point computation required.
metrics 64 vb
Coordination. DCP counters, CAS-based leases, checkpoints. Low vbucket count intentional — isolates coordination overhead from high-throughput I/O. DCP adapters use CAS leases here to distribute vbucket assignments across pods with no external coordinator.
session 256 vb
Head stores. Entity, attribute, affordance, and context stores. User accounts. Split from metrics to isolate high-throughput head store I/O. Key prefixes: ent::, surf::, cand::. L2 LMDB DaemonSet streams this bucket continuously.
